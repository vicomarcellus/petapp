Техническое задание
Веб-приложение «Трекер здоровья кота»

1. Цель
Создать простое веб-приложение для ежедневного отслеживания состояния здоровья кота.
Основные задачи:
- фиксация состояния по дням
- заметки по симптомам и лекарствам
- наглядная визуализация динамики
- выявление повторяющихся циклов ухудшения состояния

Приложение не ставит диагнозов и не даёт медицинских рекомендаций.

--------------------------------------------------

2. Платформа
- Web (desktop + mobile)
- Offline-first
- Один пользователь
- Один кот (MVP)

--------------------------------------------------

3. Основные сущности данных

3.1. DayEntry (запись дня)

Обязательные поля:
- date (YYYY-MM-DD)
- state_score (integer, 0–4)
- note (string)
- created_at (timestamp)

Дополнительные поля:
- ate (boolean) — ел ли сам
- vomit (boolean) — была ли рвота
- tremor (boolean) — была ли дрожь
- pain (boolean) — признаки боли
- meds (string) — лекарства / дозы

--------------------------------------------------

4. Шкала оценки состояния

Используется единая шкала:

0 — критично (лежит, не ест)
1 — плохо (слабость, дрожь)
2 — средне (вялый, но ест)
3 — нормально
4 — отлично

Каждому значению соответствует цвет:
0 — тёмно-красный
1 — красный
2 — жёлтый
3 — светло-зелёный
4 — зелёный

Оценка выбирается один раз в день.

--------------------------------------------------

5. Календарь состояния

- Годовая матрица формата 7×52 (как GitHub Contributions)
- Один квадрат = один день
- Цвет квадрата = state_score
- Наведение / тап:
  - дата
  - балл
  - короткая заметка

Навигация:
- переключение годов
- кнопка «Сегодня»

--------------------------------------------------

6. Экран добавления / редактирования дня

Элементы:
- дата (по умолчанию текущая)
- выбор состояния (0–4)
- чекбоксы:
  - ел сам
  - рвота
  - дрожь
  - боль
- поле «Заметка»
- поле «Лекарства»

Кнопка:
- Сохранить

--------------------------------------------------

7. Экран просмотра дня

Отображает:
- дату
- состояние (текст + цвет)
- все симптомы
- заметку
- лекарства

Кнопка:
- Редактировать

--------------------------------------------------

8. Аналитика (минимум)

- количество дней с плохим состоянием за месяц
- количество дней подряд без ухудшения
- визуальное выявление повторяющихся спадов (по календарю)

Без сложных графиков на MVP.

--------------------------------------------------

9. Хранение данных

MVP:
- localStorage или IndexedDB

Возможность расширения:
- экспорт / импорт JSON
- облачная синхронизация (не в MVP)

--------------------------------------------------

10. Нефункциональные требования

- минимальное количество кликов
- быстрый ввод данных
- читаемая цветовая схема
- отсутствие перегруженного интерфейса
